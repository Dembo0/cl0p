# Технический отчет по анализу: Linux-вариант шифровальщика Cl0p (ELF)

## 1. Введение и архитектура

Исследуемый образец представляет собой исполняемый файл формата ELF, предназначенный для работы в операционных системах семейства Linux. Программа реализует функционал шифровальщика (Ransomware)

---
## 2. Поведенческий анализ работы малвари

### Этап 1: Демонизация процесса (`main`)
При запуске программа выполняет ряд операций для скрытия своей работы в системе:
1. **Разветвление процесса:** Вызывается `__libc_fork()`. Родительский процесс немедленно завершает работу, а дочерний продолжает выполнение в фоновом режиме.
2. **Отключение от терминала:** Дочерний процесс вызывает `__setsid()`, создавая новую сессию и отвязываясь от управляющего терминала. Это предотвращает завершение процесса при закрытии сессии пользователем.
3. **Настройка окружения:** Сбрасывается маска создания файлов с помощью `__umask(0)`, а рабочая директория изменяется на корневую `/` через `__chdir("/")`.
4. **Закрытие дескрипторов:** Закрываются стандартные потоки ввода, вывода и ошибок (`stdin` [0], `stdout` [1], `stderr` [2]) с помощью `__libc_close`.
5. **Маскировка в логах:** Инициализируется запись в системный журнал с помощью `openlog("daemon-named", 17, 8)` (имя `"daemon-named"`, флаги `LOG_PID | LOG_NOWAIT`, категория `LOG_USER`). В лог пишется строка через `syslog`.

### Этап 2: Обход файловой системы (`start_attack` / `ransom_dir`)
Шифровальщик запускает процедуру сканирования целевых директорий (`start_attack`):
* `/opt` (установленное стороннее ПО)
* `/u01`, `/u02`, `/u03`, `/u04` (традиционные пути СУБД Oracle)
* `/home` (домашние папки пользователей)
* `/root` (домашний каталог администратора)

![start_attack](<data/Pasted image 20260717143526.png>)

Процедура обхода каталогов (`ransom_dir`):
1. **Чтение содержимого:** Директории открываются через `__opendir` и считываются с помощью `__readdir64`.
2. **Пропуск служебных ссылок:** С помощью `strcmp` игнорируются ссылки на текущий (`.`) и родительский (`..`) каталоги.
3. **Рекурсия:** При обнаружении подкаталога (проверка типа через маску `(stat_buf.st_mode & 0xF000) == 0x4000`) функция вызывает сама себя рекурсивно.
4. **Фильтрация:** Игнорируются файлы, в имени которых содержится строка `"C_I_0P"`, чтобы избежать повторного шифрования.

### Этап 3: Алгоритм шифрования файлов (`ransom_alg`)
1. **Блокировка доступа:** Права на файл изменяются на `0600` (`rw-------`) через вызов `__chmod`, чтобы ограничить доступ к нему во время операции.
2. **Проекция в память:** Файл открывается с помощью `__libc_open64`. Размер файла проверяется через `fxstat_wrapper`. Если он превышает 100 МБ, размер ограничивается этой величиной. Далее файл маппится в память с помощью `__mmap64` для выполнения побайтовых изменений.
3. **Генерация сессионного ключа:** В цикле создается случайный 117-байтный массив (сессионный RC4-ключ) с использованием встроенной таблицы символов и генератора случайных чисел.
4. **Шифрование данных:** Данные в памяти шифруются по алгоритму RC4 с использованием сгенерированного сессионного ключа.
5. **Сохранение сессионного ключа:** Сессионный ключ шифруется симметричным алгоритмом RC4 с использованием мастер-ключа и дописывается в конец создаваемого файла метаданных с расширением `.C_I_0P`.
6. **Освобождение памяти:** Вызывается `__munmap` для записи измененного буфера обратно на диск.

## 3. Критические архитектурные уязвимости

В архитектуре шифрования данного ELF-варианта допущены две критические ошибки:

### А. Симметричное шифрование сессионного ключа
Сгенерированный для конкретного файла 117-байтный RC4-ключ зашифровывается **симметричным алгоритмом RC4** с использованием жестко заданного в теле программы мастер-ключа:
`Jfkdskfku2ir32y7432uroduw8y7318i9018urewfdsZ2Oaifwuieh~~cudsffdsd`

Так как ключ симметричный и вшит в тело программы, любой исследователь может дешифровать файлы в обратном порядке без выплаты выкупа:
1. Расшифровать сохраненный в конце файла сессионный RC4-ключ с помощью мастер-ключа.
2. С помощью полученного сессионного ключа расшифровать содержимое самого файла.

### Б. Утечка данных из стека (Information Leak)
При сохранении зашифрованного ключа программа записывает в файл `.C_I_0P` ровно **256 байт** (`0x100`), хотя зашифрованный ключ занимает всего **117 байт** (`0x75`). Оставшиеся 139 байт заполняются неинициализированными данными из стека процесса. 

Код создания ключа:
![create_key](<data/Pasted image 20260717230541.png>)

В файл попадает структура данных `clopelfkeyfile`:
```c
struct clopelfkeyfile {
    byte encr_rc4key[117];      // Зашифрованный сессионный ключ RC4
    stat fdstat;                // Метаданные файла до шифрования (88 байт)
    long fdid;                  // Уникальный ID inode (8 байт)
    int fd;                     // Дескриптор файла (4 байта)
    int fdmappedaddr;           // Адрес проецирования памяти (4 байта)
    off_t fdsize;               // Размер файла (8 байт)
    int rc4_msize;              // Размер мастер-ключа (4 байта)
    long rc4_fsize;             // Размер сессионного ключа (8 байт)
    int fdnameaddr;             // Адрес строки имени файла (4 байта)
    int frameaddr;              // Frame pointer (4 байта)
    int retaddr;                // Адрес возврата (4 байта)
    byte fdpathaddr[3];         // Часть путей из стека (3 байта)
};
```
Благодаря этому из метаданных файла `.C_I_0P` можно извлечь исходный размер файла (`st_size`) и время изменения его статуса (`st_ctime`) до атаки.

## 6. Варианты обнаружения

1. **Поведенческий анализ:** Правила аудита ядра Linux для отслеживания массовых вызовов `chmod` (блокировка прав) и создания файлов записок вымогателей.
2. **Логирование и корреляция:** Логика выявления маскировки процесса (когда легитимное имя `"daemon-named"` запускается из нетипичных путей, например `/tmp` или `/dev/shm`).
3. **Форензик-анализ:** Скрипт проверки файлов `.C_I_0P` на аномальный размер в 256 байт и извлечение из них структуры утечки памяти (`stat` оригинального файла).
